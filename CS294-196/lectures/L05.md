# L05 Agents for Software Development

This lecture is about code as an environment, not code as a completion task. The hard parts are localization, planning, recovery, and safety. The payoff is that software work becomes something an agent can actually execute end to end.

## Summary
- Software is the substrate for most modern work, so even a small gain in developer automation has a large effect. The lecture starts by framing agents as leverage for the whole software economy. Source: `[0:03]` to `[3:25]` in `/Users/otonashi/thinking/pratchett-os/tmp/research-123/L05.transcript.txt`.
- There is a spectrum of automation: autocomplete, chat-assisted refactoring, task-specific automation, and then full autonomous software tasks. This lecture is about the last one. Source: `[4:36]` to `[5:59]`.
- Coding agents need an environment, a way to localize files, a way to plan, a way to recover from errors, and a safety story. That is the real problem statement. Source: `[9:35]` to `[10:52]`.
- Benchmarks matter because the field is full of easy toy tasks and contaminated datasets. HumanEval, MBPP, CoNaLa, SWEBench, LiveCodeBench, and Design2Code all expose different failure modes. Source: `[12:32]` to `[22:31]`.
- CodeAct, SWE-Agent, and OpenHands show the main modeling pattern: let the model act through code, then edit, run, inspect, and try again. Source: `[23:15]` to `[30:03]`.
- The conclusion is practical and slightly grim: coding agents are useful now, but the field still needs better localization, planning, recovery, and safety before they can be trusted broadly. Source: `[52:26]` to `[1:00:39]`.

## Key Concepts
- **Automation ladder**: manual coding, autocomplete, chat-based assistance, task-level automation, and full autonomous execution. Source: `[4:36]` to `[5:14]`.
- **Coding agent**: a system that edits code, runs tools, and completes software tasks rather than just finishing the next line. Source: `[5:52]` to `[8:12]`.
- **Environment**: source repos, task trackers, docs, chat tools, and servers are all part of the software worker's world. Source: `[10:52]` to `[12:04]`.
- **File localization**: finding the right files and functions is often harder than writing the code itself. Source: `[39:14]` to `[45:35]`.
- **Pass@K**: benchmark success measured by whether at least one of K generated samples passes tests. Source: `[17:45]` to `[18:25]`.
- **Leakage**: public benchmarks can overstate capability when the training mix has seen the data already. Source: `[19:15]` to `[21:18]`.
- **CodeAct**: an agent that acts by writing code and using programs instead of one API call at a time. Source: `[23:15]` to `[25:54]`.
- **SWE-Agent**: a structured edit-and-observe loop with file windows, edit commands, and tool feedback. Source: `[26:06]` to `[28:45]`.
- **Retrieval-augmented code generation**: retrieve similar code or docs, then generate from that context. Source: `[44:54]` to `[45:35]`.
- **Sandboxing**: keep the agent inside a constrained environment so it cannot damage the main system. Source: `[54:02]` to `[55:11]`.
- **Least privilege**: give the agent only the credentials and permissions it needs. Source: `[55:15]` to `[56:38]`.

## Non-Obvious Insights
- The field keeps saying "coding," but the biggest bottleneck is usually not code synthesis. It is finding the right context and file. Source: `[39:14]` to `[45:35]`.
- The benchmarks are useful precisely because they are imperfect. The lecture uses their weaknesses to show what real agents still miss. Source: `[16:57]` to `[22:31]`.
- A too-restricted agent is basically a calculator with delusions. Safety without capability is not a solution. Source: `[56:41]` to `[57:03]`.
- Good coding agents are not just better completions. They need a loop that can inspect, test, fail, and recover without collapsing into repetition. Source: `[46:04]` to `[51:47]`.
- The lecture treats safety as an engineering reuse problem. Existing code-security ideas, Docker, permissions, and auditing already cover a lot of the needed ground. Source: `[52:26]` to `[58:02]`.
