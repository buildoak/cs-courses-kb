# L02 LLM Agents: Reasoning, Acting, and Real Tasks

This lecture draws the boundary cleanly: LLM agents are not just chatbots. They are reasoning systems that can act in an environment, recover, and do work that looks like work.

## Summary
- Shunyu Yao defines an agent as an intelligent system that interacts with an environment, then narrows the ladder to text agents, LLM agents, and reasoning agents. Source: `[0:55]` to `[4:05]` in `/tmp/agent-mux-501/01KN853HZCNQYE4QJJ70Q992ZE/research-122-L02-transcript.txt`.
- The historical arc goes from rule-based chatbots to RL text agents to GPT-style prompting, then to reasoning agents where reasoning and acting start to converge. Source: `[4:08]` to `[7:59]`.
- QA gets split into computation, knowledge, and external action. The fixes are program generation, RAG, and tool use. Source: `[8:04]` to `[12:57]`.
- ReAct is the key synthesis: think, act, observe, and loop. Without reasoning, action degrades into imitation and falls apart in new states. Source: `[14:02]` to `[23:35]`.
- The lecture's larger claim is practical: LLM agents unlock digital automation, real web tasks, software work, and chemistry tasks, not just synthetic games. Source: `[43:44]` to `[52:58]`.
- Future work is framed around training, interface design, robustness, human interaction, and benchmarks that reflect real-world reliability. Source: `[54:35]` to `[1:08:27]`.

## Key Concepts
- **Agent** means an intelligent system that interacts with an environment, whether the environment is physical, digital, or conversational. Source: `[1:17]` to `[2:31]`.
- **Text agent** is an agent whose observations and actions are in language. Source: `[3:01]` to `[3:26]`.
- **LLM agent** is a text agent that uses LLMs to act. Source: `[3:41]` to `[4:05]`.
- **Reasoning agent** is the stronger version: an LLM agent that reasons before acting. Source: `[3:50]` to `[4:05]`.
- **RAG** retrieves relevant knowledge and appends it to the prompt so the model can answer with fresher or domain-specific context. Source: `[10:28]` to `[11:14]`.
- **Tool use** adds special action tokens for calculator, search, weather APIs, and other external calls. Source: `[11:14]` to `[12:26]`.
- **ReAct** interleaves thoughts, actions, and observations in one loop. Source: `[15:14]` to `[19:57]`.
- **Reasoning + acting** is the core abstraction because it unifies QA, web interaction, robotics, and software tasks. Source: `[13:36]` to `[16:54]`.
- **Digital automation** is the application class that matters most here: assistants that can file reports, write code, debug, and find papers. Source: `[43:44]` to `[45:29]`.
- **Benchmark realism** matters because pass@k and toy environments do not measure the same thing as customer service or long-horizon work. Source: `[1:02:33]` to `[1:08:27]`.

## Non-Obvious Insights
- The real abstraction is not "LLM." It is the split between reasoning and acting, then the recognition that the two feed each other. Source: `[13:36]` to `[19:57]`.
- Tool use alone is brittle. Without reasoning, the agent merely imitates an action trace and cannot adapt when the environment changes. Source: `[21:20]` to `[23:35]`.
- Real-world progress depends on task quality as much as model quality. Synthetic grids and toy games are bad proxies for actual agent work. Source: `[45:32]` to `[46:01]`.
- Better interfaces are a research problem, not just an implementation detail. The same model can perform differently depending on the action space it gets. Source: `[59:21]` to `[1:02:20]`.
- Robustness is a different metric from one-shot success. In production, the question is often "can you succeed every time?" not "can you succeed once?" Source: `[1:03:08]` to `[1:07:07]`.
