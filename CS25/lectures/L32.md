# L32 Transformers United: Behind the Scenes of LLM Pre-training

This lecture is a reminder that the best LLM is mostly cleaned-up data plus enough compute to trust the filter.

## Summary
- The lecture starts from a blunt premise: architecture matters, but data is the backbone of LLM quality. Scaling laws decide how to split compute between model size and token budget. Source: `[4:07]` to `[9:00]`.
- The scaling-law story moves from Kaplan to Chinchilla to newer data-dependent work. The lecture also flags the inference-cost blind spot in pure compute-optimal planning. Source: `[6:33]` to `[15:12]`.
- The data sources are web, GitHub code, curated corpora, and synthetic data. FineWeb, The Stack, Stack v2, and Cosmopedia are the concrete examples the speaker leans on. Source: `[15:26]` to `[23:18]`.
- Filtering is a pipeline, not a checkbox: language filtering, quality filtering, dedup, PII removal, decontamination, formatting, and repository-aware packing all matter. Source: `[23:27]` to `[37:52]`.
- Open code LLMs are now an ecosystem, but evaluation is still brittle. HumanEval, MultiPL-E, and LiveCodeBench are the response to benchmark contamination and overfitting. Source: `[38:30]` to `[49:42]`.
- Synthetic data helps, but the lecture argues for a mixture. Filtered web still matters, and tiny curated fine-tuning sets can beat huge noisy ones. Source: `[50:20]` to `[1:01:17]`.

## Key Concepts
- **Scaling laws.** Use compute to trade off model size against token count. Chinchilla shifts the optimum toward more data; newer work says the curve also depends on data quality. Source: `[6:01]` to `[15:12]`.
- **FineWeb.** A heavily filtered web corpus, presented as one of the strongest public web datasets. Source: `[16:43]` to `[17:29]`.
- **The Stack and Stack v2.** BigCode's permissive code corpora, with v2 adding more sources like issues, commits, notebooks, Kaggle, and pull requests. Source: `[17:42]` to `[18:27]` and `[31:24]` to `[31:46]`.
- **Cosmopedia.** Synthetic text built from web prompts and open models, with web samples used to steer topical diversity. Source: `[21:54]` to `[23:12]`.
- **Deduplication.** Exact dedup and near dedup, especially MinHash-style methods, are the highest-value general filters. Source: `[25:18]` to `[25:55]`.
- **PII and decontamination.** Train a detector for secrets and names, then remove benchmark overlap so eval numbers stay honest. Source: `[34:08]` to `[35:19]`.
- **Repository-aware formatting.** Keep neighboring files together, add metadata tokens, and preserve repo structure so code assistants can use context better. Source: `[35:19]` to `[37:31]`.
- **Open code release stack.** Datatrove filters, nanotron trains, LightEval evaluates. The release is meant to be inspectable, reproducible, and useful for the community. Source: `[37:32]` to `[45:36]`.
- **Code evaluation.** HumanEval and MultiPL-E are useful, but they are vulnerable to contamination; LiveCodeBench uses fresh problems released after model cutoffs. Source: `[46:43]` to `[49:42]`.
- **Fine-tuning is a different regime.** For instruction tuning or domain adaptation, small curated datasets can beat massive noisy mixtures. Source: `[59:36]` to `[1:00:10]`.

## Non-Obvious Insights
- The best filter is often the one that removes the least amount of useful data while still cutting the junk. The speaker explicitly says some intuitive filters, like repository-star thresholds, performed badly because they threw away too much. Source: `[27:18]` to `[28:27]`.
- Ablation is not optional. The lecture says you need small-model filter tests, high-signal benchmarks, and multiple seeds or you will mistake noise for improvement. Source: `[28:31]` to `[30:05]`.
- Near dedup is singled out as the biggest win because it is broad and language-agnostic. That makes it more useful than many hand-designed quality heuristics. Source: `[32:39]` to `[33:25]`.
- Synthetic data is not a replacement for web data. The lecture says Cosmopedia still improved when mixed with filtered web, which is a strong argument for hybrid training sets. Source: `[51:52]` to `[53:07]`.
- Code and text are closer than old-school ML intuition suggested. The architecture mostly stays the same; what changes are context length, inference speed, formatting, and release discipline. Source: `[38:30]` to `[45:36]` and `[55:19]` to `[56:15]`.
- Open releases are governance products. The talk treats opt-out tools, PII handling, documentation, and benchmark hygiene as part of the model, not postscript chores. Source: `[42:20]` to `[45:36]` and `[1:00:35]` to `[1:01:15]`.

## Referenced Papers
- **Kaplan scaling laws** - the original compute, model, and token tradeoff. Source: `[6:33]` to `[7:10]`.
- **Chinchilla** - revises the optimum toward more training data. Source: `[7:38]` to `[8:29]`.
- **DeepSeek LLM scaling paper** - shows that scaling behavior depends on data quality and filtering choices. Source: `[13:56]` to `[14:44]`.
- **FineWeb** - a large, highly filtered web corpus. Source: `[16:43]` to `[17:29]`.
- **The Stack** - BigCode's permissive code dataset. Source: `[17:42]` to `[18:27]`.
- **The Stack v2** - expanded code corpus with more sources and more languages. Source: `[31:24]` to `[31:46]`.
- **Cosmopedia** - synthetic text corpus generated from web-steered prompts. Source: `[21:54]` to `[23:12]`.
- **Yi** - an example of a careful web filtering pipeline. Source: `[23:59]` to `[26:09]`.
- **StarCoder / StarCoder2** - the code model family built on the Stack datasets. Source: `[27:18]` to `[37:52]`.
- **HumanEval** - the common code completion benchmark, with pass@1 as the key metric. Source: `[46:43]` to `[47:23]`.
- **MultiPL-E** - HumanEval translated to many programming languages. Source: `[47:30]` to `[47:40]`.
- **LiveCodeBench** - a contamination-resistant benchmark that only uses post-release problems. Source: `[48:31]` to `[49:42]`.
- **Lima** - tiny instruction-tuning set that can beat far larger noisy datasets. Source: `[59:36]` to `[1:00:10]`.
