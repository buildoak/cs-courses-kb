# L03: Multi-core Arch Part II + ISPC Programming Abstractions

## Summary
- ISPC is introduced as a small C-like language that exposes SPMD parallelism without forcing you to hand-write vector intrinsics.
- `program count` and `program index` define a gang of program instances; each instance runs the same logic with its own local state.
- The same work can be mapped interleaved or blocked, and both mappings preserve the same result.
- `foreach` is the real abstraction win: you declare that iterations are independent and let ISPC assign them.
- The main lesson is the split between program semantics and implementation strategy.

## Key Concepts
- **SPMD**: single program, multiple data.
- **Program count**: how many instances are running together.
- **Program index**: which instance am I?
- **Gang of instances**: the conceptual execution group ISPC creates at a call boundary.
- **Interleaved scheduling**: each instance gets every Nth iteration.
- **Blocked scheduling**: each instance gets a contiguous chunk of the loop.
- **`foreach`**: a higher-level loop form that leaves the distribution of iterations to ISPC.
- **Semantics vs implementation**: the same answer can be produced by several different execution strategies.

## Non-Obvious Insights
- The lecture keeps hammering on the same point because it matters: the program's meaning is not the same thing as the machine schedule.
- Manual interleaving and blocking are both valid, but they are still implementation choices the programmer should not have to hard-code.
- ISPC is valuable because it makes legality obvious and scheduling flexible at the same time.
- The abstraction scales because it lets the runtime adapt to different hardware widths and different core counts.
- A sequential implementation of the ISPC call would still be correct; it would just throw away the point of the abstraction.
