# L15: Domain Specific Programming Languages

## Summary
- The lecture argues for DSLs as the space between productivity and performance: narrow enough to optimize, broad enough to express the domain naturally.
- Halide is the first case study. The algorithm stays separate from the schedule, and the schedule controls loop structure, tiling, vectorization, and memory layout.
- Auto-scheduling becomes possible because the schedule space is structured enough for search, not because the compiler is guessing blindly.
- Lizst is the second case study. The compiler owns the mesh representation, ghost cells, communication, and synchronization strategy.
- The lecture ends by widening the same idea to specialized hardware, including direct FPGA generation from a high-level description.

## Key Concepts
- **Domain-specific language**: a constrained language designed around one kind of workload.
- **Algorithm vs schedule**: what to compute versus how to place and execute it on hardware.
- **Halide stage**: a functional step in an image-processing pipeline.
- **Schedule**: the set of choices that decide tiling, fusion, vectorization, and execution order.
- **Auto-scheduler**: a search system that tries schedule variants for you.
- **Mesh DSL**: a language that lets you operate on fields and relations in a mesh without exposing the raw data structure.
- **Ghost cells**: copied boundary data used to make distributed mesh execution correct.
- **Graph coloring**: a way to break conflicting updates into conflict-free phases.
- **Compiler-owned representation**: the user writes in domain terms, and the compiler chooses the actual data layout.
- **Direct hardware generation**: skip the programmable processor and emit FPGA circuits instead.

## Non-Obvious Insights
- In Halide, the schedule is often the real performance program.
- The big win from DSLs is not just convenience. It is that the compiler gets semantic facts that C/C++ cannot expose safely.
- Auto-scheduling works because the design space is small and structured enough to search.
- Lizst avoids atomics on GPUs by turning dependency conflicts into graph-coloring phases.
- A good DSL matches the user's mental model: pixels for image people, tensors for ML people, meshes for physicists.
- The same high-level description can target CPUs, GPUs, clusters, or FPGAs if the primitives are composable enough.

## Connections
- This lecture extends the course from parallel execution into compiler and language design.
