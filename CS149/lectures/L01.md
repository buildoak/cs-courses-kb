# L01: Why Parallelism? Why Efficiency?

## Summary
- The lecture frames parallel computing as a cost-model problem: speedup only matters if the data gets to the right place.
- A program is just instructions changing state, but the hardware can still exploit hidden parallelism inside that instruction stream.
- Multicore, SIMD, and hardware multithreading are three different ways to raise throughput.
- Memory latency and cache-line behavior explain why "faster compute" does not automatically mean faster programs.
- Once general-purpose cores hit power and parallelism limits, specialization becomes the obvious next move.

## Key Concepts
- **Program semantics vs implementation**: the meaning of the code is separate from how the machine runs it.
- **Registers and memory**: processors store values in registers, but the real bottleneck often comes from memory traffic.
- **Cache locality**: caches are small, low-latency storage that assume nearby data will be reused soon.
- **Multicore parallelism**: duplicate execution context so independent work can run on separate cores.
- **SIMD / vector registers**: one instruction operates on many data values at once.
- **Throughput vs latency**: the class keeps using throughput as the better mental model for modern chips.
- **Specialized hardware**: general-purpose cores are no longer the only answer when efficiency is the target.

## Non-Obvious Insights
- The hardware can find a little instruction-level parallelism, but the outer loop is where the real parallelism lives.
- Spending transistors on more clever single-thread execution has diminishing returns once dependency graphs flatten out.
- Caches are not just faster memory; they are bets about locality and reuse.
- The power wall matters as much as the transistor count. More frequency is not a free speedup path.
- The lecture's real thesis is that performance is mostly about moving data well, not just doing arithmetic fast.
