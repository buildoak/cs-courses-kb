# L10: Efficiently Evaluating DNNs on GPUs

## Summary
- The lecture opens with Assignment 3's transparent-circle renderer to show that correctness constraints come before speed.
- DNN workloads collapse into repeated convolutions, matrix multiplies, and non-linearities; the real cost is usually tensor movement, not math.
- The big wins come from fusion and tiling: keep intermediate tensors on chip, and do the next operation before you spill to memory.
- FlashAttention is the cleanest example of the same idea: chunk the softmax so the `N^2` attention matrix never has to live in memory.
- GPUs win because DNNs are highly parallel, SIMD-friendly, and high in arithmetic intensity once the kernels are written well.
- Lower precision and tensor cores are the hardware extension of the same theme: amortize overhead across more math.

## Key Concepts
- **Correctness first**: the circle renderer is parallel only if order-sensitive overlaps are handled correctly.
- **Convolution**: a local weighted stencil over pixels or activations; the lecture treats it as the core DNN primitive.
- **Matrix multiply view**: many conv layers can be reformulated as dense linear algebra.
- **Fusion**: combine adjacent operations so tensors do not bounce in and out of memory.
- **Tiling / blocking**: work on chunks that fit in cache or shared memory.
- **Attention**: a Q/K/V workload dominated by the size of the intermediate score matrix.
- **Softmax chunking**: compute max/sum incrementally so attention can be fused block by block.
- **cuDNN algorithm selection**: different convolutions want different implementations; there is no universal best kernel.
- **Tensor cores**: special matrix-multiply instructions exposed as hardware acceleration for the same workload shape.
- **Low precision**: reduced numeric width is another way to raise throughput when the workload tolerates it.

## Non-Obvious Insights
- The lecture is not really about "AI." It is about memory traffic hiding inside a compute-heavy workload.
- A better model architecture and a better kernel are separate wins. One reduces FLOPs, the other reduces bandwidth pressure.
- FlashAttention is fundamentally a locality trick, not a magical new attention algorithm.
- cuDNN's API surface is ugly on purpose because the best convolution algorithm depends on shape, stride, and layout.
- The hardware story ends up looking like a compiler story: if the system can fuse and reframe the math, it wins.
- Tensor cores are just the latest version of the old CS149 theme: move the overhead out of the hot loop.

## Connections
- This is the GPU-scale version of the locality and arithmetic-intensity arguments from the early CPU lectures.
- It mirrors the loop-fusion ideas in the Spark and cache-coherence material: do more work per trip to memory.
- The lecture explains why libraries like cuDNN, JAX, and Triton matter: they encode the kernel choices the programmer should not hand-write.
- It sets up the later discussion of specialization: if matrix multiply dominates, the hardware should look like matrix multiply.
