# L12: Memory Consistency

## Summary
- The lecture finishes cache coherence and then separates the harder question: how should reads and writes to different addresses appear across threads?
- Sequential consistency is the intuitive model: one global order, program order preserved inside each thread.
- Write buffers are the performance escape hatch, but they let reads bypass writes and therefore break sequential consistency.
- Relaxed models like TSO and PSO buy performance by weakening write/read ordering.
- Fences are the repair mechanism: they force the memory system to quiesce before the program continues.
- The practical contract is data-race-free programming: synchronize shared data and let the system relax everything else.

## Key Concepts
- **Coherence vs consistency**: coherence is per-address; consistency is about ordering across addresses.
- **Single-address serialization**: coherence says the last write to one line is the value later readers should see.
- **Sequential consistency**: all operations appear in one global order that respects program order.
- **Memory ordering**: write-read, read-read, read-write, and write-write relationships between different addresses.
- **Write buffer**: lets a store proceed without stalling every later load.
- **TSO / PSO**: weaker consistency models that permit reordering for performance.
- **Fence**: a hardware barrier that forces pending memory operations to settle.
- **Data race**: unsynchronized concurrent access where at least one access is a write.
- **DRF programming**: if code is synchronized, the memory model stays understandable even on relaxed hardware.

## Non-Obvious Insights
- Coherence does not save you from consistency bugs. A coherent system can still be semantically surprising.
- Even a machine with no caches would still need a memory consistency model.
- The performance win from write buffers is real, but it shifts complexity onto the language/runtime author and the systems programmer.
- Fences are not a design pattern. They are a cost you pay when you need to re-establish order.
- The lecture's real programming advice is not "memorize every model." It is "avoid data races."

## Connections
- This lecture is the semantic complement to the previous cache-coherence lecture.
- It matters most for synchronization libraries, compiler writers, and kernel code.
- It foreshadows language-level memory models, which are the next layer above hardware.
- The bus-serialization intuition from coherence stops being enough once multiple addresses enter the picture.
