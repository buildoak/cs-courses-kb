# L09: Distributed Data-Parallel Computing Using Spark

## Summary
- Spark is the cluster-scale version of the sequence-operator idea from the previous lecture.
- RDDs are immutable, lineage-tracked sequences; transformations build new RDDs instead of mutating old ones.
- `persist` and in-memory execution are the answer to MapReduce's repeated disk checkpoints.
- Narrow dependencies let Spark optimize partition-by-partition; wide dependencies force shuffles.
- The lecture's real claim is that fault tolerance and memory locality can coexist if the programming model carries enough structure.

## Key Concepts
- **Warehouse-scale computer**: a cluster treated like one large computer, with power, cooling, network, and failure in the model.
- **MapReduce**: the original large-cluster data-parallel abstraction the lecture uses as the starting point.
- **RDD**: resilient distributed dataset, Spark's immutable sequence abstraction.
- **Transformation**: a function from one RDD to another RDD.
- **Lineage**: the chain of transformations that lets Spark recover data instead of checkpointing every step.
- **`persist`**: keep an RDD in memory instead of recomputing it or spilling everything back to disk.
- **Narrow dependency**: one partition depends on one partition upstream.
- **Wide dependency**: one partition depends on many partitions upstream, which usually means a shuffle.
- **Shuffle**: the expensive data movement step that redistributes data across the cluster.
- **Partitioning**: the way Spark slices work so each partition can move independently when possible.

## Non-Obvious Insights
- Spark is not "MapReduce but faster." It changes the storage assumption from disk-first to memory-first while keeping fault tolerance through lineage.
- The lecture keeps returning to memory because in distributed systems the first bottleneck is often data movement, not arithmetic.
- The same sequence-operator mindset from the GPU lecture works at cluster scale because the abstraction is doing the real work.
- Persisting hot RDDs is not a convenience feature. It is the thing that makes repeated analytics loops practical.
- Narrow dependencies matter because they let Spark treat partitions independently and avoid global reshuffles.
- The lecture's optimization story is really about turning a fault-tolerant system into a cache-friendly one without losing recovery semantics.
