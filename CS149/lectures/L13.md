# L13: Fine-Grained Synchronization and Lock-Free Programming

## Summary
- The lecture opens by naming the three failure modes that matter here: deadlock, livelock, and starvation.
- It then uses cache coherence and atomic primitives to show how locks are actually built in hardware, starting with test-and-set and moving to compare-and-swap.
- Ticket locks show up as the fairness fix: enqueue with an atomic increment, spin on `now_serving`, release with one write.
- The linked-list example shows why coarse locking is safe but expensive, and why hand-over-hand locking is the more concurrent correct version.
- The lecture closes by shifting from blocking locks to lock-free programming, where progress is speculative and compare-and-swap is the retry point.

## Key Concepts
- **Deadlock**: nobody can move because each actor holds something the next actor needs.
- **Livelock**: work is happening, but no useful progress is being made.
- **Starvation**: the system keeps moving, but one thread never gets service.
- **Test-and-set**: a simple atomic read-modify-write primitive used to implement a lock.
- **Compare-and-swap**: read a value, compare it to an expected value, and update only if the comparison matches.
- **Ticket lock**: a fair lock that gives each waiter a ticket and serves them in FIFO order.
- **Coarse-grained locking**: one lock around the whole data structure.
- **Hand-over-hand locking**: hold the current node and the previous node while traversing a linked structure.
- **Lock-free programming**: do the work speculatively, then check at the end whether the world changed.
- **Blocking vs non-blocking**: blocking waits for exclusive access; non-blocking retries instead of stopping the world.

## Non-Obvious Insights
- A lock can be correct and still be a performance disaster if it serializes the whole data structure.
- Ticket locks move most contention out of the acquire path and into cheap reads while waiting.
- In a linked list, releasing `previous` too early destroys the only guarantee that `current` is still valid.
- Lock-free does not mean fair. It only means somebody keeps making progress.
- Compare-and-swap still rides on cache coherence, so the cost shifts into the cache line, not out of the system.
- The lecture treats deadlock, livelock, and starvation as debugging categories, not academic trivia.
