# L02: A Modern Multi-Core Processor

## Summary
- The lecture starts from a simple kernel and shows why instruction-level parallelism alone cannot recover the outer-loop parallelism hidden by serial C.
- Superscalar and out-of-order execution help, but dependency graphs cap how much the hardware can infer on its own.
- Multicore and SIMD are the two big scaling moves here: duplicate execution context for independent work, or widen the datapath for repeated work.
- Divergent branches waste vector lanes, so coherent code matters as much as raw compute width.
- Power, not transistor supply, is why clock speed stopped being the main performance lever.
- The lecture ends by opening the door to ISPC as a software abstraction that makes the hardware story easier to exploit.

## Key Concepts
- **Instruction-level parallelism**: the processor can sometimes overlap independent instructions inside one stream.
- **Superscalar execution**: multiple execution units on one core, fed by out-of-order scheduling.
- **Dependency graph**: the real limiter on how much parallel work hardware can safely infer.
- **Multicore execution**: duplicate registers and execution units so separate instruction streams can run together.
- **SIMD / vector width**: one instruction operates on 8 or more data elements at once.
- **Masking / predication**: keep the vector lanes busy by selectively disabling lanes that do not need work.
- **Coherent execution**: all lanes follow the same control path; divergence burns efficiency.
- **Power wall**: frequency scaling stopped because energy and heat stopped scaling politely.
- **Heterogeneous cores**: big cores for latency, small cores for efficiency and throughput.

## Non-Obvious Insights
- The lecture is not saying "more hardware is bad." It is saying hardware gets more expensive when it has to guess at parallelism.
- A processor can have plenty of transistors and still stop gaining useful speedup once the dependency graph flattens out.
- Divergence is a utilization problem, not just a control-flow problem.
- The chip design choice is really about where to spend complexity: on clever single-thread execution, or on more parallel lanes.
- The transition to ISPC is not a detour. It is the software answer to the hardware limits the lecture just spent an hour building.
