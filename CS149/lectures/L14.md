# L14: Midterm Review

## Summary
- This lecture is a review pass over the exam surface: compare-and-swap, atomic min, ticket locks, MapReduce, coherence, relaxed consistency, segmented scan, and CUDA scheduling.
- Compare-and-swap is the central atomic primitive, and atomic min is shown as a speculative loop around it.
- Ticket locks are the fairness example: a small change in the acquire path cuts contention and makes waiting FIFO.
- MapReduce is framed as a bandwidth-and-locality system, not a magic distributed-compute trick.
- The end of the lecture revisits coherence, relaxed consistency, segmented scan, and CUDA warp/thread-block scheduling as the conceptual checklist for the midterm.

## Key Concepts
- **Atomic compare-and-swap**: only update memory if the current value still matches the expected value.
- **Atomic min**: speculatively compute a minimum, then commit it only if the compare-and-swap succeeds.
- **Ticket lock**: atomic increment to get a ticket, spin on `now_serving`, release by incrementing `now_serving`.
- **MapReduce**: `map` emits key-value pairs, shuffle groups by key, `reduce` consumes key plus value list.
- **Cache coherence**: one-address ordering and visibility across caches.
- **Relaxed consistency**: different addresses can be observed in different orders by other threads.
- **Segmented scan**: a scan over a sequence of sequences.
- **CUDA bulk launch**: launch many thread blocks, each with a fixed number of threads and resources.
- **Warp**: the 32-thread execution unit the lecture uses as the mental model for NVIDIA GPUs.

## Non-Obvious Insights
- CAS is not magic. It is still a write-heavy coherence event that can get expensive under contention.
- Ticket locks reduce acquire-side contention by making the waiting path mostly read-only.
- MapReduce is strong when latency can be hidden and the real bottleneck is bandwidth.
- A 5,000-node Spark or MapReduce cluster can lose to a laptop if locality is bad enough.
- Relaxed consistency does not let a single thread violate program order; it changes what other threads can observe.
- CUDA scheduling is mostly about matching the work shape to SM occupancy, shared memory, and warp width.
