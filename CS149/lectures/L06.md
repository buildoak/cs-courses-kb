# L06: Performance Optimization II: Locality, Communication, and Contention

## Summary
- Shared-memory systems are physically messy. Rings, crossbars, cache slices, and GPUs on the same chip all make the same load/store request cost different amounts.
- The real metric is not just balance but bytes communicated per unit of work. That is arithmetic intensity.
- The lecture splits communication into inherent and artifactual. One comes from the algorithm itself; the other comes from the order and layout you choose.
- Cache blocking is the classic fix. Reuse the same cache lines before they get evicted, and you move up the arithmetic-intensity curve.
- Contention is separate from bandwidth. Average capacity can look fine while synchronized demand spikes still stall the system.
- Roofline is the diagnostic model: flat top means compute-bound, sloped side means bandwidth-bound, and the only way rightward is more work per byte or a better algorithm.

## Non-Obvious Insights
- The same computation can get dramatically faster just by walking the data in a better order.
- A cache miss is often a reuse-distance problem, not proof that the data was never loaded.
- Contention is timing-sensitive. Randomizing arrivals can improve effective bandwidth without changing the amount of work.
- More compute units force you to raise arithmetic intensity if memory stays fixed.
- Roofline is not just a pretty plot. It tells you whether to tune the algorithm or the memory traffic.
