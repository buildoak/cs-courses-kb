# L07: GPU Architecture and CUDA Programming

## Summary
- CUDA keeps the same SPMD shape as ISPC, but the programmer expresses work in thread blocks instead of task gangs.
- CPU and GPU live in separate address spaces; `cudaMemcpy` is explicit work, not an implementation detail.
- Shared memory and `__syncthreads()` are the block-local tools for reuse and correctness.
- Warps are the hardware's implicit SIMD unit; the scheduler only gets SIMD when program counters line up.
- Thread blocks are both the unit of locality and the unit of synchronization, so block size is a resource decision.

## Key Concepts
- **CUDA thread**: one instance of an SPMD program, identified by thread and block indices.
- **Thread block**: the programmer-visible unit of launch, synchronization, and shared memory.
- **Warp**: 32 threads that the GPU may execute together when their program counters match.
- **SM**: streaming multiprocessor, the hardware unit that holds many live execution contexts and schedules warps.
- **Implicit SIMD**: the GPU opportunistically runs same-PC threads together; the programmer does not issue vector ops directly.
- **Separate address spaces**: host memory and device memory are distinct unless you explicitly move data or use unified memory.
- **Shared memory**: fast per-block storage for reuse across threads in the same block.
- **`__syncthreads()`**: the block-level barrier that makes cooperative loading safe.
- **Occupancy**: how many warps / blocks can stay resident given register and shared-memory limits.
- **Atomics**: cross-thread / cross-block synchronization for shared counters and histograms.

## Non-Obvious Insights
- The block size is not just launch syntax. It is a promise that the hardware can keep the whole block live if the code uses a barrier.
- Shared memory is not mainly about capacity. It is about turning repeated neighbor loads into one cooperative load per block.
- CUDA's SPMD model is intentionally higher level than its execution model. The hardware can reorder and co-schedule warps without changing program meaning.
- Warp-level SIMD is implicit, which is why CUDA can change implementation details across generations without forcing every kernel to be rewritten.
- The dangerous boundary is not "parallel vs sequential." It is "what the block can assume about concurrency" versus "what it cannot assume about ordering."
- A program can be correct and still rejected if its barrier semantics would deadlock under the hardware's block scheduling rules.
