# L04: Parallel Programming Basics

## Summary
- ISPC turns a function call into a gang of program instances. `program count` and `program index` define the semantics; SIMD lanes are just one possible implementation.
- `foreach` is the real abstraction win. You declare independence and let the compiler decide whether the work is interleaved, blocked, or something else valid.
- The same trick shows up everywhere in the lecture: local copies first, global reduction later. That is how you avoid races on `sum` and `diff`.
- Tasks are the next layer up. You expose work, the runtime maps tasks to threads, and a thread pool beats one-thread-per-task by a wide margin.
- Amdahl's law is the hard stop. If enough of the algorithm stays serial, more parallel hardware stops helping. Sometimes the right move is a different algorithm.

## Non-Obvious Insights
- `foreach` is a contract about independence, not a promise about schedule.
- Interleaved vs blocked is a locality question, not a semantics question.
- The compiler is allowed to execute the gang sequentially and still be correct. Performance is the separate problem.
- Fine-grained tasks can make the scheduler slower than the sequential baseline.
- If the serial tail is hard, the best optimization may be to change the math, not the machinery.
