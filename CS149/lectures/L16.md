# L16: Transactional Memory 1

## Summary
- Transactional memory raises synchronization from lock choreography to an atomic, isolated, serializable region.
- The lecture treats a transaction as sequential consistency at transaction granularity, not instruction granularity.
- The main programming win is failure atomicity and composition: the whole region either commits or aborts.
- Performance depends on two policy choices: eager vs lazy versioning, and pessimistic vs optimistic conflict detection.
- With hardware support, transactions can match fine-grained locking and avoid the collapse you get from coarse locks.

## Key Concepts
- **Atomicity**: all reads and writes inside a transaction either take effect together or not at all.
- **Isolation**: uncommitted writes are invisible to other transactions.
- **Serializability**: the system can impose a valid serial order, usually by commit order.
- **Sequential consistency**: transactional memory gives the same feel, but at the level of whole transactions.
- **Read set / write set**: the values read and the values written during a transaction.
- **Conflict types**: read-write and write-write conflicts force serialization.
- **Eager versioning**: update state immediately; commit is cheap, abort is expensive.
- **Lazy versioning**: buffer writes until commit; abort is cheap, commit is expensive.
- **Pessimistic detection**: check for conflicts as accesses happen.
- **Optimistic detection**: assume no conflict, validate on commit.
- **Granularity trade-off**: coarse locks kill concurrency, fine-grained locks buy concurrency but add overhead and complexity.
- **Failure atomicity**: aborting a transaction rolls back the whole region, which is harder to get right with locks.

## Non-Obvious Insights
- The abstraction is the point: the programmer says what must be atomic, the runtime decides how to serialize it.
- The transaction order is defined by commit, which makes the model feel like one bulk memory operation.
- Transactions solve the “correct vs fast” trade-off by moving the concurrency policy below the code.
- Eager vs lazy is mostly a commit-vs-abort cost swap, not a correctness difference.
- The lecture’s TCC example matters because it shows the best case: TM can preserve easy programming without giving up fine-grained concurrency.
- The real win is composability. Transactional regions compose in a way ad hoc lock protocols usually do not.

## Connections
- Cache coherence is the prerequisite; transactions build on the same shared-memory ordering machinery.
- Database ACID is the conceptual ancestor for the semantics.
- Java `synchronized` HashMap is the blunt safe baseline; fine-grained locking is the expensive alternative.
- Tree updates on independent nodes show why transactional granularity matters.
- Lecture 17 turns these abstractions into software barriers, STM, and hardware TM.
