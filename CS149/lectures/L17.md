# L17: Transactional Memory 2

## Summary
- The lecture starts with the implementation side: transaction descriptors, transaction records, and software barriers.
- STM inserts read/write bookkeeping into the code, but the barriers add a lot of overhead and many checks are redundant.
- The McRT data shows compiler optimization can cut STM overhead hard, but the single-thread cost is still real.
- Hardware TM reuses caches and coherence: cache-line R/W bits track transactional state, and commit publishes state lazily.
- The commercial outcome is blunt: software TM survived in localized uses; hardware TM mostly did not.

## Key Concepts
- **Software barriers**: compiler or VM inserted calls that turn ordinary reads and writes into TM-aware operations.
- **Transaction descriptor**: per-thread or per-transaction metadata for the read set and write set.
- **Transaction record**: per-data metadata such as lock state or version number.
- **False conflict**: coarse object granularity makes unrelated accesses collide.
- **Version timestamps**: a global timestamp plus per-transaction timestamps support validation.
- **Undo log**: eager versioning needs a way to roll back writes on abort.
- **Optimistic read / pessimistic write**: the hybrid STM design used in the lecture.
- **Read-set validation**: at commit, validate that nothing in the read set was invalidated.
- **Cache-line R/W bits**: HTM metadata that tracks transactional read and write state.
- **Checkpointed registers**: abort requires restoring CPU state, not just cache state.
- **Lazy optimistic commit**: transactions stay private until commit, then publish via coherence.
- **Bus serialization**: the teaching model is simple because only one commit can happen at a time.

## Non-Obvious Insights
- STM is not just “slow because software.” The barriers are full of bookkeeping that a compiler can often eliminate.
- Granularity is a performance knob. Object-level tracking reduces complexity, but it can create false conflicts.
- Lazy optimistic TM is basically “use the cache as a write buffer and coherence as the commit protocol.”
- Hardware TM is not magic. It is coherence plus metadata plus rollback support.
- The bus explanation is pedagogically clean because it serializes commit, but it is not a scalable architecture.
- The practical lesson is localize TM to the hard parts, not the whole program.
