# CS149 Gems

## Top Insights
- The course's durable thesis is that parallel speed comes from moving less data, in the right shape, at the right time. That shows up as cache locality, arithmetic intensity, and avoiding intermediate materialization from CPU caches to FlashAttention. [L01](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L01.md:24), [L06](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L06.md:35), [L10](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L10.md:26), [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:37)
- The strongest abstraction pattern is "state the legality, hide the schedule." ISPC, `foreach`, tasks, scans, RDDs, Halide schedules, and transactional regions all let the runtime or compiler own mapping decisions once the semantic contract is clear. [L03](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L03.md:23), [L04](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L04.md:29), [L08](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L08.md:28), [L09](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L09.md:25), [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:24), [L16](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L16.md:25)
- The "best" algorithm is machine-shaped, not just asymptotically elegant. Static chunking can beat dynamic scheduling, a work-efficient scan can lose on a warp, and CUDA blocks only work when the hardware can keep the whole block live. [L05](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L05.md:34), [L08](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L08.md:23), [L07](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L07.md:23), [L14](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L14.md:27)
- Synchronization is a layered stack, not one problem. Coherence, consistency, locks, lock-free CAS, and transactional memory reuse the same substrate but make different promises about visibility, ordering, and progress. [L11](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L11.md:25), [L12](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L12.md:23), [L13](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L13.md:27), [L16](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L16.md:30), [L17](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L17.md:27)
- Specialization is the end of the arc, not a side quest. DSLs and hardware accelerators work when the abstraction is narrow enough to compile into a better schedule, a better layout, or even a better machine. [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:24), [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:28), [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:28), [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:30)

## Surprising Connections
- Spark lineage and cache coherence are the same trick at different scales: keep local views valid while allowing parallel updates, then pay for wider visibility only when needed. [L09](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L09.md:23), [L11](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L11.md:24), [L12](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L12.md:23)
- FlashAttention, cache blocking, RDD persistence, Halide fusion, and double buffering all exist to avoid materializing intermediates and to do more work per byte moved. [L06](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L06.md:31), [L09](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L09.md:26), [L10](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L10.md:26), [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:23), [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:27)
- Segmented scan is the hidden bridge between sparse matrix work, ragged collections, and graph-style pipelines. It turns irregular shape into regular operator structure. [L08](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L08.md:25), [L08](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L08.md:26), [L09](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L09.md:25)
- Ticket locks, work stealing, and transactional memory all convert contention into a structured protocol: queue, steal, or retry instead of letting the whole system thrash. [L05](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L05.md:36), [L13](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L13.md:24), [L16](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L16.md:28), [L17](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L17.md:27)
- CUDA blocks, Spark partitions, and Halide schedules all make locality a scheduling unit. The machine gets faster when the unit of work matches the unit of data reuse. [L07](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L07.md:24), [L09](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L09.md:27), [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:25), [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:31)

## Under-Appreciated Ideas
- Divergence is mainly a utilization problem, not a pure control-flow problem. Wide hardware can sit idle while the code is still correct. [L02](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L02.md:25), [L07](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L07.md:26)
- Coherence does not save you from consistency bugs, and even a cache-free machine still needs a memory model. [L12](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L12.md:23), [L12](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L12.md:24)
- `foreach` is a legality contract, not a speed guarantee. The compiler can execute the gang sequentially and still be correct. [L03](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L03.md:23), [L04](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L04.md:31)
- Dynamic scheduling is not automatically better. If the work unit is too small, the scheduler tax beats the balance benefit. [L05](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L05.md:34), [L05](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L05.md:35)
- Fences are costs to re-establish order, not a default design pattern. [L12](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L12.md:26)
- Auto-scheduling works because the search space is structured enough to search, not because the compiler magically understands the domain. [L15](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L15.md:25)
- Pipelining is near-free performance only if you can afford the extra buffering it needs. [L18](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L18.md:27)
- Hardware TM is basically coherence plus metadata plus rollback. The novelty is the contract, not the substrate. [L17](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L17.md:27), [L17](/Users/otonashi/thinking/building/cs-courses-kb/CS149/lectures/L17.md:28)
