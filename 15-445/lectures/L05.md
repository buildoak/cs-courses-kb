# L05 Database Storage: Log-Structured Merge Trees & Tuples

This lecture is about the storage decisions that make tuple movement possible and make lookup performance sane: log-structured storage, record IDs, heap files, index-organized storage, and the ugly corners around big values.

## Summary
- The lecture distinguishes OLTP from OLAP before getting into structures: transactional workloads need fast inserts and updates, analytical workloads need fast scans. Source: `8:44 - 10:47`.
- Log-structured storage replaces in-place updates with an in-memory memtable plus on-disk SSTables, then uses compaction and bloom filters to keep lookup costs tolerable. Source: `24:56 - 39:16`.
- Deletes become tombstones, not immediate removal, so the system can preserve append-only behavior and later materialize the deletion in compaction. Source: `38:36 - 39:16`.
- Physical tuple handles matter: row IDs, CTIDs, and record IDs let engines move tuples or compact pages without breaking references. Source: `14:13 - 17:30`, `18:31 - 20:22`, `46:14 - 52:23`.
- Index-organized storage makes the index leaf page the data page, which is great for reads but more awkward for updates and compaction. Source: `50:40 - 54:54`.
- The lecture ends by warning that large values and overflow pages push the storage layer toward extra indirection, and it tees up row vs column storage for next class. Source: `1:15:26 - 1:21:48`.

## Key Concepts
- **OLTP / OLAP.** The lecture uses workload shape to motivate the storage design. Source: `8:44 - 10:47`.
- **Memtable.** The in-memory write buffer that absorbs updates before they are flushed. Source: `25:59 - 27:06`.
- **SSTable.** The on-disk sorted run that receives flushed log-structured data. Source: `26:24 - 27:06`.
- **Compaction.** The merge process that removes redundant versions and reduces lookup cost. Source: `26:39 - 27:06`, `36:20 - 39:16`.
- **Bloom filters.** Skip hints that tell the engine whether a key might exist in an SSTable level. Source: `36:20 - 37:44`.
- **Tombstones.** Delete markers that preserve append-only semantics until compaction. Source: `38:36 - 39:16`.
- **Record IDs / CTIDs.** Physical locators that encode where a tuple lives inside the storage engine. Source: `14:13 - 17:30`, `46:14 - 52:23`.
- **Heap files.** Unordered page collections that still support tuple movement through page metadata and slot arrays. Source: `43:08 - 44:46`, `50:40 - 54:54`.
- **Index-organized storage.** A B+ tree layout where the leaves store the data itself. Source: `29:55 - 34:49`, `50:40 - 52:23`.
- **Overflow pages.** Auxiliary pages for values that do not fit inline. Source: `1:15:26 - 1:16:20`.
- **Record identity vs logical identity.** The engine keeps a physical address for itself, while the application keeps logical keys. Source: `7:24 - 8:07`, `16:46 - 17:08`.

## Non-Obvious Insights
- LSM trees do not eliminate update cost. They defer it into compaction and lookup filtering.
- Bloom filters are not location indexes. They are skip hints, which is why the lecture uses them to avoid whole SSTable levels.
- Record IDs are a deliberate escape hatch: stable enough for higher layers, physical enough for the storage engine.
- Index-organized storage is not just "an index with data in the leaves." It changes update cost, compaction, and tuple identity.
- The page abstraction is already getting leaky by the time the lecture introduces big values and overflow pages.
