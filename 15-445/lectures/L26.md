# L26 BigQuery + Snowflake + Redshift + Databricks + DuckDB

This lecture is the systems survey: same SQL, different bets on storage, shuffles, caches, and code generation.

## Summary
- BigQuery descends from Dremel: shared-disk storage, vectorized query processing, PAX-like columnar layout, inverted indexes, and an explicit shuffle phase. Source: `[32:52]` to `[39:09]`.
- Snowflake is a managed OLAP system that leans on shared-disk storage, aggressive compute-side caching, push-based vectorized execution, and precompiled primitives. Source: `[39:38]` to `[45:47]`, `[49:39]` to `[51:50]`.
- Redshift is Amazon’s warehouse answer in the same cloud OLAP space, but the lecture treats it as a more classic shared-nothing system with SQL on top. Source: `[46:02]` to `[48:18]`.
- Databricks Photon is the response to JVM friction: a C++ engine with pull-based vectorized execution, precompiled primitives, and expression fusion. Source: `[1:03:44]` to `[1:08:26]`.
- DuckDB and MotherDuck are the counterpoint to warehouse gigantism: local-first analytics with optional cloud offload when the query or data outgrows the laptop. Source: `[1:20:22]` to `[1:24:43]`.

## Key Concepts
- **Dremel.** Google’s internal system that became BigQuery. Source: `[32:52]` to `[34:20]`.
- **BigQuery.** Google’s managed OLAP service on top of the Dremel architecture. Source: `[33:59]` to `[34:20]`.
- **Shuffle phase.** The explicit repartitioning boundary that separates pipeline stages and enables fault tolerance and adaptive scaling. Source: `[35:50]` to `[37:27]`.
- **In-memory shuffle.** BigQuery’s service layer for holding intermediate results between stages. Source: `[36:22]` to `[37:27]`.
- **Shared-disk storage.** Compute and storage are disaggregated; the warehouse pays for object storage and caches aggressively on compute nodes. Source: `[39:55]` to `[40:18]`.
- **Columnar storage / PAX.** BigQuery’s internal layout is described as feeling like Parquet or ORC, but using PAX-style column storage. Source: `[35:01]` to `[35:15]`.
- **Inverted index.** BigQuery’s extra index type for text and JSON-like access. Source: `[35:18]` to `[35:24]`.
- **Adaptive query execution.** BigQuery can change the plan at runtime once the shuffle exposes the real data shape. Source: `[35:38]` to `[35:44]`, `[38:51]` to `[39:09]`.
- **Push-based vectorized execution.** Snowflake’s execution style, inherited from earlier vectorized systems and combined with operator primitives. Source: `[41:05]` to `[41:30]`.
- **Precompiled primitives.** Snowflake and Photon both specialize operator code ahead of time instead of generating the whole plan dynamically every time. Source: `[41:28]` to `[42:30]`, `[1:05:15]` to `[1:06:24]`.
- **Expression fusion.** Photon fuses predicates inside a filter instead of fusing whole operator pipelines vertically. Source: `[1:07:16]` to `[1:08:26]`.
- **MotherDuck.** A cloud offload layer for DuckDB queries. Source: `[1:23:02]` to `[1:24:17]`.

## Non-Obvious Insights
- BigQuery’s shuffle is not just a repartitioning hack. It is also the fault-tolerance checkpoint and the place where the scheduler can rebalance stragglers.
- Snowflake wins by hiding old ideas, not replacing them. The architecture still depends on caches, columnar storage, and query compilation; it just packages them as a managed service.
- Photon’s design shows the practical limit of code generation: the faster engine is the one the team can keep changing, not the one with the fanciest compiler story.
- The real product choice is not “SQL or no SQL.” It is where the system puts the boundary between persistent data, cached data, and recomputable intermediate results.
- DuckDB/MotherDuck makes the local-first case explicit: many workloads do not need warehouse-scale infrastructure, only a clean way to burst into the cloud when necessary.

## Connections
- To L25: this is the same shuffle/data-movement story, but now attached to actual products instead of abstract architectures.
- To vectorized execution: BigQuery, Snowflake, and Photon all depend on the same batch-processing core from earlier lectures.
- To query planning: the systems differ mainly in how much planning happens globally, how much is cached, and how much gets recompiled at runtime.
