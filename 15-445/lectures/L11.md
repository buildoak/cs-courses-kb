# L11 Index Concurrency Control

This lecture is not about transactions. It is about keeping internal data structures physically correct while multiple threads hit them at the same time.

## Summary
- Physical correctness means pointers still land on real pages; logical correctness means transactions see the right visible data. Source: `[6:23]` to `[8:14]`.
- Latches protect data structures, locks protect transactions. Latches are short-lived, decentralized, and should not depend on the OS when avoidable. Source: `[9:34]` to `[13:23]`, `[15:07]` to `[17:44]`.
- Test-and-set spin latches are the simplest primitive, but they burn CPU and fall apart under contention or NUMA traffic. Source: `[18:32]` to `[23:43]`.
- Futex-backed mutexes are already a user-space spin path plus an OS fallback, which is why standard mutexes are not "just a system call." Source: `[24:01]` to `[27:12]`.
- Hash tables can latch by page or slot and avoid deadlock because the traversal direction is fixed. B+ trees need latch crabbing because splits and merges can invalidate the path. Source: `[29:01]` to `[40:12]`.
- Optimistic latch coupling assumes splits are rare, takes the fast path, and retries only when the tree proves it was wrong. Source: `[47:15]` to `[52:10]`.

## Key Concepts
- **Physical correctness.** A pointer you follow must still point to a real page or node, or the structure can break even if the transaction logic is fine. Source: `[7:30]` to `[8:14]`.
- **Logical correctness.** A transaction should still see the right user-level state, but that is a later concern. Source: `[6:48]` to `[7:23]`.
- **Latch vs lock.** Latches protect threads and data structures for tiny critical sections; locks protect transactions for much longer intervals. Source: `[9:34]` to `[11:20]`.
- **Spin lock.** A test-and-set loop is fast when uncontended, but it wastes CPU when the latch is busy. Source: `[18:32]` to `[20:18]`.
- **NUMA cost.** Repeated cache-line bouncing across sockets is one reason spin-heavy designs get ugly fast. Source: `[20:03]` to `[21:41]`.
- **CAS / test-and-set.** The CPU gives you an atomic compare-and-swap primitive, and that is the foundation for user-space latches. Source: `[21:53]` to `[23:43]`.
- **Futex.** The user-space mutex path spins first, then falls back to the kernel only when contention forces it. Source: `[24:01]` to `[25:25]`.
- **Hash-table latching.** Page-level latching is coarse but simple; slot-level latching gives more concurrency at the cost of more metadata. Source: `[29:27]` to `[34:12]`.
- **B+ tree latch crabbing.** Hold the parent while you find the child, then release ancestors once the child is safe. Source: `[36:45]` to `[41:47]`.
- **Release order.** Release from the top down to maximize parallelism, even when either order is logically safe. Source: `[41:05]` to `[41:47]`.
- **Optimistic coupling.** Assume splits and merges are rare, descend with read latches, and restart only when you must split or merge. Source: `[47:15]` to `[52:10]`.
- **Leaf scan deadlocks.** Sibling-pointer scans can deadlock when threads move in opposite directions and each holds a latch the other needs. Source: `[53:04]` to `[55:43]`.
- **Retry policy.** The right answer is usually to abort the local attempt, release what you hold, and try again instead of trying to reason about the whole tree while blocked. Source: `[55:45]` to `[1:02:10]`.

## Non-Obvious Insights
- The data structure determines the concurrency protocol. Hash tables can safely release behind themselves; B+ trees cannot because the path can change under them.
- The cheapest latch path is the uncontended one. That is why the lecture spends so much time on "what happens when no one is fighting you?"
- "Kill yourself and retry" sounds wasteful, but in this regime it is usually cheaper than building a heavier-weight scheduler inside the index.
- The optimistic path only works because high fan-out makes split and merge events rare enough to bet on.
- The root latch is a bottleneck, so releasing it early is not an optimization detail. It is the difference between a real concurrent index and a fake one.

## Connections
