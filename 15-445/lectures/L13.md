# L13 Query Execution Part 1

This lecture is about the operator contract: how a query plan actually moves tuples when you pretend everything is an iterator.

## Summary
- Query execution starts with a simple promise: each operator can call `next` on its child and keep the plan composable, no matter whether the child is a scan, index lookup, network source, or subquery. Source: `[11:33]` to `[12:32]`.
- The Volcano/pipeline model turns that promise into a practical execution style, with `open`, `next`, and `close` controlling iterator state and lifecycle. Source: `[12:33]` to `[12:57]`.
- Hash join is the concrete example for why pipeline breakers exist: the build side must finish before the probe side can produce correct output. Source: `[13:07]` to `[16:24]`.
- Materialization, vectorized batch processing, and the pull/push control-flow choices are different ways to move data through the same operator graph. Source: `[20:17]` to `[35:34]`.
- The lecture also shows how the same execution contract extends to operator fusion, index scans, updates, inserts, and cached/materialized results. Source: `[35:46]` to `[58:34]`.

## Key Concepts
- **Iterator model.** `next` is the core API. Parents do not care how children produce tuples as long as they can keep calling `next`. Source: `[11:33]` to `[12:04]`.
- **`open` / `close`.** Operators keep state explicitly. `open` creates the iterator, `close` tears it down. Source: `[12:04]` to `[12:32]`.
- **Volcano model.** The lecture names this iterator-based processing model and treats it as the standard baseline for most DBMS execution engines. Source: `[12:33]` to `[12:57]`.
- **Pipeline.** A pipeline is a sequence of operators that can stream tuples through without blocking on intermediate materialization. Source: `[12:57]` to `[13:07]`, `[15:56]` to `[16:24]`.
- **Pipeline breaker.** Hash join is the clearest example: the build side blocks the probe side until the hash table is complete. Source: `[14:03]` to `[15:23]`.
- **Materialization model.** Operators emit full tuple batches or full result sets instead of one tuple at a time. Source: `[20:17]` to `[20:57]`, `[24:27]` to `[25:36]`.
- **Vectorized model.** Operators work on batches of tuples, which is the more common modern compromise between tuple-at-a-time and full materialization. Source: `[26:46]` to `[29:42]`.
- **Pull model.** The root calls down through children and pulls tuples up the tree. Source: `[30:58]` to `[31:41]`.
- **Push model.** Operators push tuples upward, with a scheduler deciding when each stage runs. Source: `[31:41]` to `[34:03]`.
- **Operator fusion.** If operators sit in the same pipeline, the engine can wire them together and avoid unnecessary materialization. Source: `[23:04]` to `[23:57]`, `[34:52]` to `[35:34]`.
- **Index scan and DML operators.** The same interface covers index scans, delete/update paths, and insert paths. Source: `[48:27]` to `[58:34]`.

## Non-Obvious Insights
- Inference: The point of the iterator API is not elegance. It is interchangeability. Once everything is `next`-driven, the parent can compose a child from anywhere.
- Inference: Pipeline breakers are not an implementation detail. They are the physical boundary where the engine must stop pretending the data can stay fully streaming.
- Inference: Vectorization is not just "bigger batches." It changes the CPU/cache behavior enough that the execution model becomes part of performance, not just syntax.
- Inference: Push and pull are scheduling choices on top of the same operator contract. The tree shape does not change, but the runtime behavior does.
- Inference: The same execution boundary that serves joins also serves DML, indexing, and result reuse. Query execution is a shared substrate, not a join-only topic.
