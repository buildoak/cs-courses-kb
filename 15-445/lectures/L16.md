# L16 Timestamp Ordering & Optimistic Concurrency Control

This lecture bets on the no-conflict path. The database lets transactions run, then uses timestamps and validation to decide who really gets to commit.

## Summary
- Timestamp ordering assigns timestamps to transactions and tuples so the system can impose a logical order without taking a heavyweight lock on every access. Source: `[2:41]` to `[5:04]`.
- OCC is the optimistic version of that idea: transactions do their work in a private workspace, then switch to validation only when they want to commit. Source: `[11:14]` to `[12:29]`.
- Validation can look backward or forward. The lecture keeps it simple and uses serial validation plus forward validation to decide whether a commit is safe. Source: `[23:05]` to `[27:18]`.
- The commit edge is the synchronization point. The system compares read and rewrite sets against other transactions and aborts if the ordering would break serializability. Source: `[25:02]` to `[30:57]`.
- OCC works when conflicts are rare. If the workload is hot, the abort cost eats the advantage fast. Source: `[1:02:43]` to `[1:04:20]`.

## Key Concepts
- **Logical timestamp.** A monotonic integer that stands in for ordering. It is not wall-clock time. Source: `[4:23]` to `[5:04]`.
- **Timestamped objects.** Tuples can carry metadata about the last transaction that created or updated them. Source: `[3:28]` to `[4:18]`.
- **Read phase.** Transactions can read and write locally without coordinating on every step. Source: `[11:14]` to `[11:48]`.
- **Validation phase.** The system checks whether the private work conflicts with transactions that are already done or still active. Source: `[11:50]` to `[12:24]`, `[23:05]` to `[24:52]`.
- **Write phase.** If validation passes, the private changes are installed into the global database. Source: `[12:21]` to `[12:29]`.
- **Read set / rewrite set.** The validation logic compares the objects a transaction read and the objects it rewrote. Source: `[24:07]` to `[25:17]`.
- **Forward validation.** Compare the committing transaction against active transactions that might observe stale data. Source: `[25:02]` to `[27:18]`.
- **Backward validation.** Check whether committed transactions already wrote data this transaction should have seen. Source: `[24:17]` to `[24:52]`.
- **Repeatable reads.** Copying objects into the private workspace keeps subsequent reads stable within the transaction. Source: `[11:17]` to `[11:48]`.

## Non-Obvious Insights
- OCC does not remove coordination. It postpones it until the commit boundary.
- The timestamp is a policy signal, not a clock. The system cares about ordering, not actual elapsed time.
- Validation is cheap when conflicts are rare and painful when they are not. That is the whole trade.
- The private workspace is not just an implementation detail. It is what makes repeatable reads possible without blocking every access.
- The lecture is really about moving the cost curve. Locking pays continuously; OCC pays in bursts.

## Connections
- To `L17`: MVCC turns the timestamp idea into durable version chains instead of a single private workspace.
- To locking lectures: pessimistic protocols block early; optimistic protocols validate late.
- To recovery: commit order still has to become durable later. OCC only handles correctness before the crash.
- To the rest of the engine: the validation logic depends on the execution engine being able to track read and rewrite sets precisely.
