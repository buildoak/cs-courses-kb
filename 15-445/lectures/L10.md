# L10 Vector Indexes, Inverted Indexes, Filters, Tries

This lecture is a data-structure dump with a point. The point is not "what index is best" but "what query semantics does each structure actually support."

## Summary
- Bloom filters are cheap existence tests, not location lookups, so they work best as prefilters before a richer index. Source: `[2:40]` to `[5:56]`, `[11:17]` to `[12:44]`.
- Counting Bloom filters add deletes, cuckoo filters compress the fingerprint path, and SURF extends the idea to range queries for immutable workloads like SSTable compaction. Source: `[8:41]` to `[12:24]`.
- Skip lists are probabilistic, in-memory structures. They buy simpler maintenance at the cost of being awkward once disk pages and reverse scans matter. Source: `[14:40]` to `[17:18]`, `[25:10]` to `[27:41]`.
- Tries and radix trees compress shared prefixes. They cut space and lookup work, but they can still produce prefix-level false positives after compression. Source: `[29:28]` to `[41:24]`.
- Inverted indexes solve exact term search; vector indexes solve semantic search with embeddings and approximate nearest-neighbor queries. Source: `[42:48]` to `[1:02:48]`.

## Key Concepts
- **Filter vs index.** A filter answers "could it be here?" while an index answers "where is it?" Source: `[2:56]` to `[3:06]`.
- **Bloom filter.** Multiple hashes map a key into a bitset. The structure guarantees no false negatives, but it can return false positives. Source: `[4:01]` to `[5:57]`.
- **Counting Bloom filter.** Replace bits with counters so deletes become possible, but memory cost rises. Source: `[8:36]` to `[9:30]`.
- **Cuckoo filter.** Store a fingerprint instead of the whole key, making membership tests cheaper and more compact. Source: `[9:51]` to `[10:48]`.
- **SURF.** A compact, immutable filter that can answer exact membership and range membership after compaction. Source: `[10:50]` to `[12:24]`.
- **Skip list.** Multiple linked-list layers created by coin flips; each higher layer skips about half the keys below it. Source: `[14:40]` to `[18:09]`.
- **Delete in skip lists.** Mark logically deleted first, then physically remove top-down when it is safe. Source: `[25:13]` to `[27:10]`.
- **Trie.** Keys are decomposed into digits or bits; the path from root to leaf reconstructs the key. Source: `[29:28]` to `[31:58]`.
- **Radix tree / Patricia tree.** Compress repeated vertical and horizontal runs in a trie to reduce memory and lookup cost. Source: `[34:37]` to `[36:24]`.
- **Inverted index.** A dictionary maps terms to posting lists, so the engine can jump straight to matching record IDs. Source: `[44:02]` to `[46:27]`.
- **Full-text systems.** Lucene, Elasticsearch, Tantivy, and Postgres GIN all use variations on dictionary + posting-list layouts. Source: `[46:29]` to `[53:03]`.
- **Vector index.** Embeddings turn text into floating-point arrays, then approximate nearest-neighbor search finds "close enough" semantics. Source: `[55:28]` to `[57:05]`.
- **Vector filtering.** Metadata predicates like year or tenant often have to be combined with vector search, which creates a planning problem. Source: `[57:42]` to `[1:00:14]`.
- **ANN families.** The lecture sketches IVF-style clustering and navigable small-world graphs as the two common families. Source: `[58:46]` to `[1:01:30]`.

## Non-Obvious Insights
- The lecture is really about query shape. Exact membership, prefix matching, term search, and semantic search are different problems, so one "index" cannot solve all of them.
- Filters are not a poor man's index. They are a separate optimization layer that can sit in front of one.
- Skip lists look simpler than B+ trees only until the storage model stops being pure memory. Then page boundaries and reverse scans make the abstraction leak.
- Trie compression is not free. Once you compress paths, you may need to re-check the actual tuple because the prefix can match more than the full key.
- Vector search changes the meaning of correctness. The answer is no longer "exactly this row" but "the nearest rows according to an embedding model."
