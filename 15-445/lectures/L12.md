# L12 Sorting & Aggregation Algorithms

This lecture is about squeezing sorting and grouping through the memory wall. Once disk is real, the algorithm is mostly a pass-planning problem.

## Summary
- Sorting is not only for `ORDER BY`. It is also a way to make aggregation and top-N queries cheaper later in the plan. Source: `[6:23]` to `[9:51]`.
- Top-N heap sort gives you a one-pass answer when you only need the first few ordered rows. Source: `[10:21]` to `[13:49]`.
- External merge sort is the core disk-spilling algorithm: make sorted runs in memory, write them out, then merge pass by pass until the whole dataset is ordered. Source: `[14:22]` to `[29:17]`.
- Double buffering hides I/O by overlapping the next read with the current write, which keeps the CPU from sitting idle on disk latency. Source: `[35:29]` to `[37:19]`.
- Comparison cost matters. Code specialization and prefix comparisons can shave a lot of wasted work from large sorts. Source: `[38:08]` to `[41:31]`.
- Hash aggregation usually wins when order does not matter, but sorting can still be the better answer when it also satisfies `ORDER BY` or `DISTINCT` for free. Source: `[48:36]` to `[1:02:48]`.

## Key Concepts
- **Early materialization.** Carry the full tuple through the plan when you know you will need it later. Source: `[16:14]` to `[18:23]`.
- **Late materialization.** Carry only a record ID while sorting, then fetch the tuple later if enough rows survive. Source: `[16:48]` to `[18:23]`.
- **Top-N heap sort.** Use a bounded heap to keep only the best rows seen so far. Source: `[10:21]` to `[13:49]`.
- **External merge sort.** Use pass zero to create sorted runs, then merge them recursively into larger runs. Source: `[14:43]` to `[29:17]`.
- **Memory budget.** Sorting depends on how many buffer frames you get, and real systems expose that as a query-level knob. Source: `[21:24]` to `[22:24]`.
- **Run count and cost.** The lecture walks through the pass count and IO cost as `1 + log_B(N)`-style growth, with a concrete page-count example. Source: `[28:32]` to `[35:25]`.
- **Double buffering.** Split the memory pool so one half writes while the other half reads and sorts the next batch. Source: `[35:29]` to `[37:19]`.
- **Comparison specialization.** Hardcode the comparison for the concrete data type instead of paying a function-pointer call on every compare. Source: `[38:08]` to `[40:49]`.
- **Prefix comparison.** For long strings, compare a prefix first and only fall back to the full string if needed. Source: `[40:51]` to `[41:31]`.
- **Clustered index reuse.** If the index already stores rows in the desired order, do not sort the data again. Source: `[41:46]` to `[47:52]`.
- **Sorting for aggregation.** `DISTINCT` and some `GROUP BY` patterns can be implemented as a sorted scan that drops duplicates. Source: `[48:36]` to `[51:09]`.
- **Hash aggregation.** Build a hash table keyed by the group-by columns, then update a running aggregate per key. Source: `[51:19]` to `[53:29]`, `[1:00:23]` to `[1:01:47]`.
- **External hashing.** Partition first, aggregate second; use one hash to fan out the data, then a second hash table that fits in memory. Source: `[53:37]` to `[59:24]`.
- **Recursive hashing.** If skew collapses too many rows into one partition, do another repartitioning round. Source: `[1:00:03]` to `[1:00:16]`.

## Non-Obvious Insights
- Sorting is often worth doing even when the query does not ask for sorted output, because the same order can make later operators cheap.
- Late materialization only wins when enough rows die early. If most rows survive, you just pay the fetch cost twice.
- The bottleneck is not only algorithmic. Compare-call overhead and data movement can matter as much as the sort itself.
- Sorting and hashing are not rivals in the abstract. They are two different ways to manufacture locality from an unordered stream.
- The lecture is really about reducing random I/O. Everything else is downstream of that constraint.
