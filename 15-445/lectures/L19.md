# L19 Database Recovery with ARIES

This lecture turns crash recovery into a protocol problem. ARIES is the answer: log enough, repeat history on restart, and make undo itself restartable.

## Summary
- ARIES is the canonical recovery protocol for making a database safely recoverable, and it is designed to survive crashes during recovery itself. Source: `[5:12]` to `[5:58]`.
- Write-ahead logging is the core rule: flush the log before you flush dirty pages, then replay updates in order after a crash. Source: `[6:10]` to `[7:28]`.
- LSNs, page LSNs, and flush LSNs give the system a single ordering relation between log records and page contents. Source: `[8:07]` to `[11:37]`.
- Recovery uses three phases: analyze, redo, and undo. The redo phase repeats history; the undo phase logs its own work through CLRs. Source: `[8:23]` to `[8:35]`, `[42:01]` to `[48:27]`.
- Fuzzy checkpoints are the practical trick: capture transaction and dirty-page metadata without stopping the world. Source: `[29:49]` to `[38:21]`.

## Key Concepts
- **ARIES.** The recovery protocol the lecture treats as the canonical implementation for crash-safe databases. Source: `[5:12]` to `[5:24]`.
- **Write-ahead logging (WAL).** Log records must reach durable storage before the corresponding page is written out. Source: `[6:10]` to `[6:34]`.
- **LSN.** A monotonically increasing log sequence number that orders log records. Source: `[8:07]` to `[9:38]`.
- **Page LSN.** The last LSN that modified a page. It tells the system whether the page is newer or older than the flushed log. Source: `[10:01]` to `[11:10]`.
- **Flush LSN.** The highest LSN known to be durable in the log. Source: `[10:31]` to `[11:02]`.
- **Transaction table.** The set of active transactions and their most recent LSNs. Source: `[33:30]` to `[34:47]`.
- **Dirty page table.** The pages that were dirty when the checkpoint started, plus the log record that first dirtied each page. Source: `[35:07]` to `[35:24]`.
- **Fuzzy checkpoint.** A begin/end log pair that snapshots recovery metadata while transactions continue running. Source: `[37:19]` to `[38:21]`.
- **CLR.** Compensation log records that record undo work so recovery can restart after partial rollback. Source: `[21:34]` to `[23:08]`, `[57:11]` to `[58:01]`.
- **Undo-next-LSN.** A pointer that lets recovery jump directly to the next log record that still needs to be undone. Source: `[22:55]` to `[23:08]`, `[57:11]` to `[57:39]`.
- **Repeat history.** Redo does not guess which updates reached disk; it replays the log history and lets page LSNs skip already-applied work. Source: `[6:37]` to `[7:28]`, `[46:23]` to `[47:07]`.

## Non-Obvious Insights
- Redo is intentionally dumb about durability. It repeats history and uses page LSNs to decide whether a page already reflects a log record. Source: `[6:37]` to `[7:28]`, `[46:23]` to `[47:07]`.
- Checkpoints are metadata snapshots, not stop-the-world barriers. The begin/end markers let recovery reconstruct a consistent view while transactions keep running. Source: `[37:19]` to `[38:21]`.
- Undo has to be restartable too. CLRs make partial rollback durable, so a crash during recovery can resume from the last finished undo step. Source: `[28:49]` to `[29:04]`, `[55:55]` to `[58:01]`.
- The dirty page table stores the first log record that dirtied a page, not the latest one, because redo needs the earliest safe restart point. Source: `[35:07]` to `[35:24]`, `[46:23]` to `[47:07]`.
- Strong isolation makes the log order easier to reason about. The lecture assumes strong 2PL / serializable ordering so recovery can focus on log replay, not concurrent write reordering. Source: `[44:16]` to `[46:18]`.
- The big design win is failure containment. ARIES is built so the system can crash during recovery and still come back cleanly. Source: `[5:47]` to `[5:58]`, `[57:55]` to `[58:01]`.
