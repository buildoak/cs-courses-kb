# L25 Distributed Analytical Databases

This lecture is about one thing: analytics wins by moving less data, not by pretending the network is free.

## Summary
- ETL and ELT both funnel operational data into a warehouse so analysts can query one place instead of a pile of silos. Source: `[4:17]` to `[5:53]`.
- Star schemas deliberately duplicate dimension data to keep joins shallow; snowflake schemas save space but push you into deeper joins. Source: `[8:06]` to `[12:34]`, `[14:39]` to `[15:03]`.
- Distributed OLAP keeps the same logical query model as a single-node database, but it adds coordination around shuffles, intermediate data, and node failure. Source: `[15:17]` to `[20:35]`, `[31:30]` to `[34:27]`.
- MapReduce is presented as a new name for old exchange/shuffle ideas, and the lecture is blunt that distributed OLAP systems like Vertica do it better. Source: `[21:38]` to `[23:31]`.
- The real boundary is not SQL versus no SQL. It is persistent data versus intermediate data, and whether the system can push compute toward the data instead of dragging petabytes across the network. Source: `[24:10]` to `[28:34]`.

## Key Concepts
- **Data warehouse / OLAP.** The centralized analytic store fed from transactional systems for cross-silo reporting and analysis. Source: `[4:17]` to `[5:53]`.
- **ETL.** Extract, transform, load: reshape data before it lands in the warehouse. Source: `[4:26]` to `[4:48]`.
- **ELT.** Extract, load, transform: land the raw data first and do the transformation inside the warehouse. Source: `[5:07]` to `[5:53]`.
- **Fact table.** The large table of repeated events, like purchases. Source: `[9:02]` to `[10:07]`.
- **Dimension table.** The smaller metadata tables that enrich the fact table. Source: `[9:29]` to `[10:12]`.
- **Star schema.** A fact table with one-hop joins out to dimension tables. Source: `[9:02]` to `[10:21]`.
- **Snowflake schema.** A normalized star schema with multi-level dimension joins. Source: `[11:05]` to `[12:34]`.
- **Column store.** An analytical layout that became important before vectorized execution made the CPU side faster. Source: `[10:24]` to `[10:30]`.
- **Vectorized execution.** Batch-oriented execution that keeps analytical operators fed efficiently. Source: `[10:24]` to `[10:30]`.
- **Shared-nothing.** Each node owns its own compute and storage; scaling means copying and repartitioning. Source: `[15:17]` to `[16:17]`, `[26:07]` to `[26:17]`.
- **Shared-disk.** Nodes share persistent storage, so they can pull the data they need locally. Source: `[29:22]` to `[29:41]`.
- **Shuffle / exchange.** The intermediate layer that re-partitions data between pipeline stages. Source: `[19:54]` to `[20:35]`, `[21:24]` to `[21:35]`.
- **Persistent data.** The immutable base data, often sitting in object storage. Source: `[19:07]` to `[19:15]`, `[24:14]` to `[24:24]`.
- **Intermediate data.** The query-generated results that move between nodes during execution. Source: `[19:26]` to `[19:35]`, `[24:46]` to `[24:52]`.
- **Predicate pushdown.** Pushing filters into object storage or storage services like S3 Select. Source: `[27:05]` to `[28:24]`.

## Non-Obvious Insights
- Star schemas are not a modeling accident. They are a deliberate denormalization trade: duplicate dimension values so the warehouse can answer with fewer joins.
- The hard part of distributed OLAP is not algebra. It is placement: which fragment of data reaches which node before the next pipeline stage fires.
- A shuffle is not just a tax on the plan. It is the boundary that makes repartitioning, fault tolerance, and later-stage parallelism possible.
- MapReduce felt new because it changed the packaging, not because it invented the idea. The lecture frames it as a rebranding of old parallel-database exchange logic.
- Object storage blurs the old push/pull line, but the core rule stays the same: move code to data when you can, and move as little data as possible when you cannot.

## Connections
- To the earlier execution lectures: the shuffle is the distributed version of the exchange operator.
- To query planning: the cost model now has to care about network movement, partitioning, and replication, not just disk I/O.
- To the next lecture: BigQuery, Snowflake, Redshift, Databricks, and DuckDB are productized versions of the same design space.
