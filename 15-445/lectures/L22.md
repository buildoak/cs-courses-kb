# L22 Introduction to Distributed Databases

This lecture is the question of where the bytes live. Once data, time, and failure are all spread across machines, every query becomes a routing decision.

## Summary
- Distributed databases differ from parallel databases because the distance and failure model changes the cost of every message. The lecture keeps coming back to one question: do you push the query to the data, or pull the data to the query? Source: `[3:40]` to `[4:18]`, `[17:37]` to `[18:18]`.
- Partitioning is the main control surface. The lecture walks through table, vertical, horizontal, hash, range, and predicate partitioning, then turns the partition key into the thing that decides whether a query stays local or fans out. Source: `[30:25]` to `[41:01]`.
- Replication improves availability and read throughput, but it adds sync policy, replica selection, and primary-versus-replica semantics. Source: `[48:38]` to `[53:18]`.
- Consistent hashing reduces rebalance churn, but it does not remove the cost of membership change. The lecture contrasts it with DynamoDB-style hierarchical replication. Source: `[46:17]` to `[49:31]`.
- The guest talk at the end makes the same point in vector search: Cassandra/Astra vector indexes still have to choose between partition-based and graph-based locality, and HNSW is fast but memory hungry. Source: `[1:08:06]` to `[1:12:58]`.

## Key Concepts
- **Distributed DB.** A database where communication cost and failure modes dominate because nodes are not physically close enough for the parallel-database assumptions to hold. Source: `[3:40]` to `[4:18]`.
- **Shared-nothing.** Each node owns its own CPU, memory, and disk, and the cluster coordinates over the network. Source: `[14:24]` to `[16:45]`.
- **Partitioning.** The rule that decides which node owns which rows, columns, or predicates. Source: `[30:25]` to `[41:01]`.
- **Partition key.** The column that determines routing. If you pick badly, you get scatter/gather and hot partitions. Source: `[39:44]` to `[41:01]`.
- **Hash partitioning.** Good for even spread, bad when membership changes or the workload is skewed. Source: `[38:22]` to `[40:51]`.
- **Range partitioning.** Useful when key order matches the query pattern. Source: `[38:45]` to `[39:10]`.
- **Predicate partitioning.** Manual sharding by value class or workload class. Source: `[38:45]` to `[39:10]`.
- **Replication factor.** The number of copies kept for availability and read scaling. Source: `[48:38]` to `[48:58]`.
- **Consistent hashing.** A rebalance strategy that minimizes data movement when partitions are added or removed. Source: `[46:17]` to `[48:18]`.
- **Hierarchical replication.** The newer DynamoDB-style answer to the same rebalance problem. Source: `[49:21]` to `[49:31]`.
- **Primary replica.** The node that receives writes first and propagates them to replicas. Source: `[15:01]` to `[15:49]`.
- **Read replica.** A replica that can take read load off the primary when the consistency model allows it. Source: `[18:11]` to `[18:57]`.
- **Clock sync.** Distributed systems have to agree on time more often than people expect. The lecture calls out Spanner and CockroachDB here. Source: `[1:03:52]` to `[1:04:53]`.
- **HNSW.** The graph-based ANN index the guest talk uses as the starting point for vector search. Source: `[1:08:58]` to `[1:10:27]`.
- **Product quantization.** The compression trick used to keep ANN search practical once the index no longer fits cleanly in memory. Source: `[1:11:27]` to `[1:12:58]`.

## Non-Obvious Insights
- Locality is the only free lunch. If the query can stay on the node that owns the data, the rest of the stack gets simpler immediately. Source: `[17:37]` to `[18:18]`, `[30:25]` to `[41:01]`.
- Replication does not remove partitioning. It multiplies the places where partitioning and routing rules have to stay consistent. Source: `[48:38]` to `[53:18]`.
- Consistent hashing solves churn, not skew. Hot keys still exist, and the lecture treats skew as the real operational tax. Source: `[46:17]` to `[48:18]`, `[50:51]` to `[51:46]`.
- Clock sync is not decorative. Spanner pays for atomic clocks; CockroachDB approximates the same problem with hybrid logical clocks. Source: `[1:04:08]` to `[1:04:53]`.
- Vector search in Cassandra is just the same cluster-layout problem with a different metric and a worse memory bill. The index is still arguing about partitioning, locality, and how much state can stay hot. Source: `[1:08:06]` to `[1:12:58]`.
