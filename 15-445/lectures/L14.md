# L14 Query Execution Part 2

This lecture is about making the execution contract run on more than one worker without breaking the query result.

## Summary
- The lecture starts from the single-worker assumption and shows why parallel query execution matters on modern hardware and across distributed systems. Source: `[1:13]` to `[4:26]`.
- Process-per-worker and thread-per-worker are the two practical execution models, with the thread model trading OS isolation for lower coordination overhead. Source: `[7:14]` to `[13:41]`.
- Parallelism is not one thing. The lecture breaks it into inter-query, intra-query, horizontal, vertical, bushy, and operator parallelism. Source: `[17:01]` to `[22:27]`.
- Exchange operators are the barrier and reshuffling primitive that let a plan cross worker boundaries through gather, distribute, and repartition steps. Source: `[23:33]` to `[30:01]`.
- IO parallelism, scheduling, and partitioning limits explain why more workers do not automatically mean faster queries. Source: `[30:01]` to `[43:20]`.

## Key Concepts
- **Worker.** The lecture uses `worker` as the generic unit, covering threads, processes, or other execution units. Source: `[2:16]` to `[2:34]`.
- **Process per worker.** Each worker is a separate OS process with its own address space and scheduler treatment. Source: `[7:14]` to `[10:33]`.
- **Thread per worker.** A lighter model where the DBMS owns scheduling more directly, while the workers share address space. Source: `[12:29]` to `[13:41]`.
- **In-query parallelism.** One query is split across multiple workers. Source: `[17:01]` to `[18:25]`.
- **Inter-query parallelism.** Multiple queries run at the same time. Source: `[17:01]` to `[18:19]`.
- **Horizontal parallelism.** The same operator runs on different partitions of the same data. Source: `[20:25]` to `[22:19]`.
- **Vertical parallelism.** Different operators or pipelines run at the same time. Source: `[20:35]` to `[22:19]`.
- **Bushy parallelism.** Distinct branches of the plan execute in parallel and then rejoin later. Source: `[21:58]` to `[22:27]`.
- **Exchange operator.** The barrier that gathers outputs, repartitions data, or shuffles it across workers. Source: `[23:47]` to `[29:08]`.
- **Gather.** Multiple inputs collapse to one output stream. Source: `[25:06]` to `[25:24]`, `[27:52]` to `[27:58]`.
- **Distribute / repartition.** One input is split across multiple outputs, or multiple inputs are repartitioned into multiple outputs. Source: `[28:18]` to `[29:08]`.
- **IO parallelism.** Parallelism can also come from multiple disks or storage devices, not just CPU workers. Source: `[33:15]` to `[39:18]`.
- **Partitioned execution.** Work is assigned to partitions or shards so each worker can make progress independently. Source: `[41:04]` to `[43:20]`.

## Non-Obvious Insights
- Inference: Parallelism is a correctness-preserving transformation, not a different query. The SQL result stays the same while the physical execution gets rearranged.
- Inference: The exchange operator matters more than the worker count. Bad exchange points can erase the benefit of having more cores.
- Inference: Scheduling policy can dominate raw parallelism. If the dispatcher is bad, the workers just wait faster.
- Inference: IO parallelism is a reminder that CPU cores are not the only bottleneck. The storage topology often decides the ceiling.
- Inference: Vectorized execution and parallel execution are separate axes. A system can use both, and many do.
