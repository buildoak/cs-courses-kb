# L06 Row vs. Column Storage + Compression

This lecture flips the storage model sideways. Row stores are convenient for updates; column stores are built for scans, compression, and analytics.

## Summary
- OLAP is the workload category that justifies column stores: analytical queries scan lots of data, need high QPS, and care about freshness and low latency. Source: `8:44 - 10:47`, `13:03 - 18:24`.
- DSM and related column-store ideas are old, but they only became mainstream when analytics workloads made the payoff obvious. Source: `22:00 - 27:24`.
- Pure column stores keep each column contiguous, usually with fixed offsets or row IDs to stitch tuples back together. Source: `28:38 - 29:26`, `31:02 - 34:01`.
- PAX is the compromise layout: row groups for locality, column chunks inside each group for scan efficiency. Source: `35:56 - 39:03`.
- Compression is part of the design, not a later optimization: dictionary encoding, delta encoding, run-length encoding, and bit packing all change what the engine can answer without decompressing. Source: `49:53 - 1:09:55`.
- The StarTree flash talk turns the lecture into a production system: Pinot splits realtime and offline paths, backs data with S3, and serves user-facing analytics at high QPS. Source: `1:12:17 - 1:20:58`.

## Key Concepts
- **OLTP vs OLAP.** The lecture uses workload shape to motivate column storage. Source: `8:44 - 10:47`.
- **Data freshness.** Analytics can be valuable only if the data is current enough to matter. Source: `13:03 - 18:24`.
- **Column store.** Store each attribute together so scans over one column do not drag unrelated fields along. Source: `22:00 - 29:26`.
- **Row store.** Keep whole tuples together when update-heavy workloads matter more than scan-heavy ones. Source: `31:02 - 34:01`.
- **DSM.** The old decomposition storage model that the lecture treats as the ancestor of modern column storage. Source: `26:56 - 27:13`, `30:56 - 31:05`.
- **PAX.** Partition rows into row groups, then split each group into column chunks. Source: `35:56 - 39:03`.
- **Fixed-length offsets.** One way to stitch tuples back together across columns without storing a tuple ID per value. Source: `28:38 - 29:26`.
- **Dictionary encoding.** Replace repeated values with smaller codes and sometimes answer queries directly on the dictionary. Source: `1:07:05 - 1:10:47`.
- **Run-length encoding.** Compress repeated values by storing a value plus a count. Source: `1:05:03 - 1:06:34`.
- **Bit packing.** Store values with just enough bits, then read them back with bit arithmetic. Source: `1:00:15 - 1:02:19`.
- **Delta encoding.** Store differences from a baseline instead of the full value each time. Source: `1:06:12 - 1:06:53`.
- **Null handling.** Row stores usually use a null bitmap; column stores often use sentinel values or per-attribute markers. Source: `1:10:14 - 1:14:54`.
- **StarTree / Pinot.** A real-time OLAP system that uses the same storage ideas to serve dashboards and user-facing analytics. Source: `1:13:03 - 1:20:58`.

## Non-Obvious Insights
- Column stores win because compression and scan locality compound each other.
- PAX is a sober compromise. Many systems advertise column storage but really use PAX-like layouts.
- Sorted dictionary codes let some predicates answer directly on the encoded domain, which avoids decoding entirely.
- Null handling is a storage-layout decision, not just a logical one.
- Compression is query planning by another name: the right scheme depends on how the predicate will touch the data.
