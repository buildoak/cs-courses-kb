# L17 Multi-Version Concurrency Control

MVCC takes the timestamp idea and makes it physical. Keep multiple versions, let readers pick a snapshot, and stop blocking on the hot path.

## Summary
- MVCC assigns begin and end timestamps to each version so the engine can decide what is visible to a transaction's snapshot. Source: `[13:25]` to `[15:40]`.
- Writers do not mutate in place. They create a new version, update the version chain, and leave the old version available to older snapshots. Source: `[14:38]` to `[16:03]`.
- Readers and writers stop fighting over the same copy. That is the main reason MVCC feels faster than single-version concurrency control. Source: `[3:19]` to `[4:21]`, `[9:23]` to `[10:30]`.
- Snapshot isolation is the practical payoff, but the lecture is blunt that snapshot isolation alone is not full serializability. Source: `[26:33]` to `[27:17]`.
- MVCC reaches into indexes, garbage collection, and page layout. It is an engine-wide design, not a storage-layer trick. Source: `[34:06]` to `[35:44]`, `[54:22]` to `[58:19]`.

## Key Concepts
- **Begin timestamp.** Assigned when a transaction starts so it knows which snapshot it is allowed to read. Source: `[13:53]` to `[14:10]`.
- **End timestamp.** Marks when a version stops being visible to later transactions. Source: `[14:24]` to `[15:08]`.
- **Visibility window.** The pair of begin/end timestamps defines which versions a transaction can see. Source: `[13:39]` to `[15:40]`.
- **Version chain.** Each logical tuple points to a chain of physical versions. Readers walk the chain until they find the right snapshot. Source: `[30:39]` to `[31:40]`, `[34:06]` to `[35:24]`.
- **Transaction status table.** Needed so the engine can tell whether a version belongs to an active, committed, or aborted transaction. Source: `[16:03]` to `[16:15]`.
- **Snapshot isolation.** A transaction sees a consistent snapshot of the database, even while other transactions keep writing. Source: `[6:00]` to `[6:35]`, `[26:33]` to `[27:17]`.
- **Primary vs secondary indexes.** Primary-key lookups can often point at the version chain directly; secondary indexes are harder because they have to cope with multiple logical and physical versions. Source: `[54:22]` to `[58:19]`, `[1:02:28]` to `[1:03:32]`.
- **Garbage collection.** Old versions can only be reclaimed once no active snapshot needs them. Source: `[1:04:08]` to `[1:06:45]`.

## Non-Obvious Insights
- MVCC does not remove concurrency-control complexity. It moves the complexity into visibility rules and storage metadata.
- The index is part of the concurrency story. If the index cannot find the right version, the snapshot guarantee breaks.
- Garbage collection is gated by active readers, not just by committed writers.
- Snapshot isolation is useful because it removes many read-write conflicts, but it still leaves some anomalies on the table.
- Version chains make reads cleaner, but they make writes and indexing less trivial.

## Connections
- To `L16`: OCC validates at commit with timestamps; MVCC uses timestamps to decide which version a transaction may even see.
- To `L18`: recovery has to preserve version metadata and keep the log consistent with the version chain.
- To query execution: scan and lookup operators now need snapshot-aware version resolution.
- To storage layout: MVCC changes how tuples, indexes, and garbage collection interact with the buffer pool.
