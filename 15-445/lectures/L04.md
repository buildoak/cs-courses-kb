# L04 Database Storage: Files & Pages

This lecture moves from the logical database view to the physical disk view. The main shift is simple: tuples stay usable only if the engine keeps enough indirection around them to move, compact, and reconstruct them.

## Summary
- The semester is now in the storage stack: disk, buffer pool manager, access methods, execution, and planning are layered systems. Source: `0:30 - 4:00`.
- The buffer pool manager is the bridge between disk and memory. It tracks page IDs, maps them to physical locations, and hands out page pointers only after the page is fetched into memory. Source: `21:06 - 23:13`.
- Pages are the unit of storage, but a database page is not the same thing as a hardware page. The lecture makes that separation explicit before it introduces page IDs and file layouts. Source: `30:02 - 41:18`.
- Slotted pages solve tuple movement and variable-length layout by storing offsets in a slot array instead of baking tuple locations into the page body. Source: `1:04:23 - 1:06:03`.
- The Neon flash talk shows the same abstraction at cloud scale: compute and storage are separated, WAL is streamed to page servers, and pages are reconstructed on demand. Source: `1:09:05 - 1:14:39`.

## Key Concepts
- **Disk-oriented architecture.** The system assumes primary storage lives on nonvolatile storage and moves pages into memory when needed. Source: `5:04 - 5:32`.
- **Buffer pool manager.** The layer that decides what lives in memory, what gets fetched, and when dirty pages are evicted. Source: `21:06 - 23:13`.
- **Page directory.** A mapping from page IDs to physical locations, similar in spirit to a translation lookaside buffer. Source: `21:42 - 22:17`, `45:11 - 46:54`.
- **Record IDs / row IDs / CTIDs.** Physical locators that encode file, page, and slot, giving higher layers a stable handle even if tuples move. Source: `7:15 - 8:36`, `14:13 - 15:35`.
- **Page metadata.** Page headers carry checksums, page IDs, page type, and free-space information. Source: `30:30 - 35:03`, `57:59 - 59:57`.
- **Heap files.** Unordered collections of pages are a standard way to represent table storage. Source: `43:08 - 44:46`.
- **Fixed page size.** Most systems choose a single page size per file, usually 4, 8, 16, or 64 KiB depending on engine. Source: `38:29 - 41:18`.
- **Free space map.** Systems track free slots per page so inserts do not have to scan every page. Source: `47:21 - 48:10`, `52:39 - 55:24`.
- **Slotted page layout.** The slot array grows from the front, tuple bodies grow from the back, and the page fills in the middle. Source: `1:04:23 - 1:05:20`.
- **Overflow pages.** Large values that do not fit inline can spill to auxiliary pages, with pointers left in the base tuple. Source: `1:15:26 - 1:16:20`.
- **Logical vs physical identity.** External keys stay logical; record IDs are internal bookkeeping. Source: `7:24 - 8:07`, `16:46 - 17:08`.

## Non-Obvious Insights
- Page IDs exist so the database can move pages without breaking references. That is the point of the indirection layer.
- The lecture keeps separating "what the application sees" from "what the storage engine does" because recovery and compaction always punch holes in naive abstractions.
- A single tuple per page is easy to reason about and bad for space; a contiguous packed page is space-efficient and bad for updates. Slotted pages are the compromise.
- The Neon talk is the same idea pushed outward: if storage can reconstruct pages from WAL and page images, eviction stops being a write-back problem.
- Page format choices are not just performance choices. They decide how much metadata the engine must preserve for later recovery.
