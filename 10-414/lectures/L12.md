# L12 Hardware Acceleration + GPUs

Lecture 12 takes the same reuse logic from CPU tiling and drops it into GPU land, where the extra constraints are SIMT execution, thread-block cooperation, and memory hierarchy management.

## Summary
- GPUs trade CPU flexibility for massive parallel throughput: lots of simple workers, fewer control decisions. Source: transcript `[1:30]` to `[6:48]`.
- The CUDA model organizes work into threads, blocks, and grids, with each thread carrying its own index into the global problem. Source: transcript `[7:05]` to `[13:29]`.
- Host code has to allocate device memory, copy data across PCIe, launch kernels, and copy results back, but real applications keep data on the GPU as long as possible. Source: transcript `[16:01]` to `[20:43]`.
- Shared memory is the key reuse layer inside a block: threads cooperatively fetch overlapping data once, synchronize, and then reuse it locally. Source: transcript `[24:47]` to `[28:55]`.
- GPU matrix multiplication is built from the same tiling logic as the CPU version, but now the tradeoff includes occupancy, register pressure, and shared-memory size. Source: transcript `[29:40]` to `[41:26]`.

## Key Concepts
- **SIMT.** Single instruction, multiple threads: threads execute the same kernel code but can diverge in their data access. Source: transcript `[7:05]` to `[9:04]`.
- **Thread/block/grid hierarchy.** Threads form blocks, and blocks form the launch grid. Source: transcript `[8:23]` to `[9:04]`.
- **Global thread index.** `blockDim.x * blockIdx.x + threadIdx.x` is the standard way to map a kernel thread to an element. Source: transcript `[9:29]` to `[13:29]`.
- **Device memory.** `cudaMalloc` and `cudaMemcpy` move data to and from GPU memory. Source: transcript `[16:01]` to `[18:22]`.
- **Shared memory.** Block-local memory enables cooperative reuse among threads in the same block. Source: transcript `[23:05]` to `[28:55]`.
- **Register tiling.** Each thread computes a small `V x V` tile in registers. Source: transcript `[29:40]` to `[31:58]`.
- **Shared-memory tiling.** A block loads an `L x L` tile once, then threads reuse it for many register-level updates. Source: transcript `[32:38]` to `[36:20]`.
- **Occupancy tradeoff.** More registers or more shared memory per block means fewer active threads or blocks. Source: transcript `[37:08]` to `[39:40]`.
- **Cooperative fetching.** Threads divide the shared-memory load so each thread pulls a subset of the tile. Source: transcript `[40:20]` to `[41:31]`.

## Non-Obvious Insights
- The lecture's first real GPU lesson is not "use more threads". It is "keep the data on the GPU and avoid PCIe round trips." Source: transcript `[18:25]` to `[20:43]`.
- Shared memory matters because neighboring threads often want overlapping input windows. The reuse opportunity is not theoretical; it is literal duplication of loads. Source: transcript `[24:47]` to `[28:55]`.
- Register tiling and shared-memory tiling are not alternatives. They are stacked layers, each exploiting reuse at a different scope. Source: transcript `[29:40]` to `[36:20]`.
- The hard part of GPU tuning is balancing the knobs against each other. Bigger tiles help reuse, but they reduce how many threads fit concurrently. Source: transcript `[37:08]` to `[39:40]`.
- The final performance boosters are mostly about access regularity: continuous reads, bank-conflict avoidance, pipelining, warp-level work, and tensor cores. Source: transcript `[42:01]` to `[43:55]`.

## Connections
- L11's tiling logic survives intact on GPU; the only change is the additional hierarchy of threads, blocks, shared memory, and registers. Source: transcript `[29:40]` to `[41:26]`.
- L10's convolution story can be lowered to the same reuse problem once the operator is expressed as matrix multiplication or windowed access. Source: transcript `[25:12]` to `[28:55]`; L10 transcript `[1:03:50]` to `[1:06:49]`.
- The GPU lecture is the clearest systems bridge in the course: it translates abstract linear algebra into concrete memory traffic and launch geometry. Source: transcript `[7:05]` to `[20:43]`.
- The rest of the course inherits this idea: performance comes from how data moves, not just from the algebra on top. Source: transcript `[42:01]` to `[44:17]`.
