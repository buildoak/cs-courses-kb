# L23 Model Deployment

Lecture 23 is the "the model is not shipped until it survives the target environment" lecture. It starts with deployment constraints, then walks through inference engines and machine learning compilation.

## Summary
- Deployment starts with the environment, not the model: size limits, no Python, embedded or mobile hardware, and app integration all shape the design. Source: transcript `[0:00]` to `[7:37]`.
- The practical export path is learned model -> computational graph + weights -> backend format such as ONNX, CoreML, or TFLite. Source: transcript `[8:11]` to `[11:04]`.
- Inference engines run that graph with limited op support, memory preallocation, and reuse; operator fusion and lower precision are where the wins come from. Source: transcript `[14:45]` to `[17:03]`.
- Machine learning compilation lifts the problem into IR modules, high-level graph transforms, lowering to loop IR, and code generation. Source: transcript `[17:10]` to `[30:55]`.
- Auto-tuning closes the loop by benchmarking generated variants, training a cost model, and searching for good kernels automatically. Source: transcript `[31:00]` to `[42:49]`.

## Key Concepts
- **Deployment constraints.** Model size, runtime availability, and hardware backends decide what can be shipped. Source: transcript `[2:00]` to `[6:58]`.
- **Model export.** A learned module has to be translated into a graph format that a backend recognizes. Source: transcript `[9:08]` to `[11:04]`.
- **Inference engine.** TensorRT, TFLite, ARM Compute Library, and CoreML are treated as graph interpreters that execute a fixed inference workload. Source: transcript `[7:45]` to `[9:00]`.
- **Dynamic graph limits.** Python-style graph construction is flexible for training, but awkward when the deployment target wants a static representation. Source: transcript `[11:11]` to `[14:16]`.
- **Memory reuse.** Inference engines preallocate scratch space and reuse activations instead of allocating fresh buffers each time. Source: transcript `[15:46]` to `[16:24]`.
- **Operator fusion.** Backends fold multiple operations into one kernel to reduce memory traffic and overhead. Source: transcript `[16:24]` to `[17:03]`.
- **IR module.** The compiler view treats a model as a module of interdependent functions over tensor values. Source: transcript `[19:40]` to `[21:24]`.
- **Loop transformations.** Split, reorder, and thread binding turn high-level tensor ops into backend-shaped loops. Source: transcript `[33:45]` to `[37:59]`.
- **Cost model.** Benchmark data becomes training data for a learned estimator that predicts which kernel variant will run best. Source: transcript `[39:15]` to `[40:52]`.

## Non-Obvious Insights
- Deployment is mostly translation and constraint management, not model math. The hardest part is getting the learned object into the right runtime shape. Source: transcript `[10:44]` to `[16:58]`.
- Dynamic Python graphs are a strength during model construction and a weakness during export. The same flexibility that helps you build the model makes backend portability harder. Source: transcript `[11:11]` to `[14:16]`.
- Inference engines and ML compilers overlap, but they differ in where their operators come from: prewritten library kernels versus generated code. Source: transcript `[17:22]` to `[30:43]`.
- The real runtime wins are memory reuse, fusion, and scheduling. "Run it on GPU" is not the interesting part. Source: transcript `[15:46]` to `[17:03]` and `[32:05]` to `[38:01]`.
- The cost model turns optimization into a data problem. The lecture's search loop is benchmark first, learn from the measurements, then search less blindly. Source: transcript `[39:15]` to `[42:49]`.
